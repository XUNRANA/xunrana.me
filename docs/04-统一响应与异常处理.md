## 1. 为什么需要统一响应格式

### 1.1 没有统一格式的问题

如果每个接口返回格式不一样，前端开发者会崩溃：

```json
// 接口 A 返回
{"data": [...], "success": true}

// 接口 B 返回
{"result": {...}, "code": 0, "msg": "ok"}

// 接口 C 返回
{"articles": [...], "total": 100}

// 接口 D 出错时返回
{"error": "Not Found", "status": 404}

// 接口 E 出错时返回
{"message": "文章不存在", "errorCode": 2001}
```

前端需要为每个接口写不同的解析逻辑，维护成本极高。

### 1.2 统一格式的好处

```json
// 成功响应 - 统一格式
{
  "code": 200,
  "message": "操作成功",
  "data": { ... }           // 具体数据（类型灵活）
}

// 错误响应 - 统一格式
{
  "code": 2001,
  "message": "文章不存在",
  "data": null
}
```

前端只需要一个统一的响应拦截器：

```javascript
// 前端 axios 拦截器（伪代码）
axios.interceptors.response.use(response => {
  const { code, message, data } = response.data;
  if (code === 200) {
    return data;              // 成功：直接返回 data
  } else if (code === 401) {
    router.push('/login');    // 未登录：跳转登录页
  } else {
    ElMessage.error(message); // 其他错误：显示错误信息
    return Promise.reject(message);
  }
});
```

## 2. Result\<T\> 统一响应设计

### 2.1 类定义

```java
@Data
@NoArgsConstructor
@AllArgsConstructor
@Builder
public class Result<T> implements Serializable {

    private int code;       // 状态码
    private String message; // 提示信息
    private T data;         // 数据（泛型，灵活适配不同类型）
}
```

### 2.2 为什么用泛型 \<T\>？

泛型让 `Result` 可以包装任何类型的数据：

```java
Result<LoginVO> loginResult;        // data 是 LoginVO 类型
Result<PageResult<ArticleVO>> page; // data 是分页结果
Result<List<CategoryVO>> list;      // data 是列表
Result<Void> deleteResult;          // data 为 null（无返回数据）
```

如果不用泛型，要么 data 是 Object（丢失类型信息），要么需要为每种返回类型创建不同的 Result 类。

### 2.3 静态工厂方法

```java
// ===== 成功响应 =====

// 无数据返回（如删除成功）
public static <T> Result<T> success() {
    return Result.<T>builder()
            .code(200)
            .message("操作成功")
            .build();
}
// 使用：return Result.success();
// 结果：{"code": 200, "message": "操作成功", "data": null}

// 有数据返回
public static <T> Result<T> success(T data) {
    return Result.<T>builder()
            .code(200)
            .message("操作成功")
            .data(data)
            .build();
}
// 使用：return Result.success(loginVO);
// 结果：{"code": 200, "message": "操作成功", "data": {"accessToken": "...", ...}}

// 自定义消息 + 数据
public static <T> Result<T> success(String message, T data) {
    return Result.<T>builder()
            .code(200)
            .message(message)
            .data(data)
            .build();
}

// ===== 错误响应 =====

// 使用 ErrorCode 枚举
public static <T> Result<T> error(ErrorCode errorCode) {
    return Result.<T>builder()
            .code(errorCode.getCode())
            .message(errorCode.getMessage())
            .build();
}
// 使用：return Result.error(ErrorCode.ARTICLE_NOT_FOUND);
// 结果：{"code": 2001, "message": "文章不存在", "data": null}

// 自定义 code + message
public static <T> Result<T> error(int code, String message) {
    return Result.<T>builder()
            .code(code)
            .message(message)
            .build();
}
```

### 2.4 为什么使用 Builder 模式？

`@Builder` 注解生成的构建者模式让对象创建更清晰：

```java
// Builder 模式（清晰、可读）
Result.<Void>builder()
    .code(200)
    .message("操作成功")
    .build();

// 等价的构造函数方式（参数多了容易搞混顺序）
new Result<>(200, "操作成功", null);
```

### 2.5 为什么实现 Serializable？

```java
public class Result<T> implements Serializable {
```

`Serializable` 接口标记该类可以被序列化（对象 → 字节流 → 对象）。虽然在 Spring MVC 中 JSON 序列化由 Jackson 处理（不要求 Serializable），但实现这个接口是好习惯：
1. 如果 Result 对象需要存入 Redis（Redis 序列化需要）
2. 如果需要跨 JVM 传输（如 RMI、分布式缓存）
3. 编码规范要求所有 POJO 实现 Serializable

## 3. ErrorCode 错误码枚举

### 3.1 设计思路

```java
@Getter
@AllArgsConstructor
public enum ErrorCode {

    // ===== 通用错误码（HTTP 标准） =====
    SUCCESS(200, "操作成功"),
    BAD_REQUEST(400, "请求参数错误"),
    UNAUTHORIZED(401, "未登录或Token已过期"),
    FORBIDDEN(403, "权限不足"),
    NOT_FOUND(404, "资源不存在"),
    INTERNAL_ERROR(500, "服务器内部错误"),

    // ===== 用户模块错误码（1xxx） =====
    USERNAME_EXIST(1001, "用户名已存在"),
    USER_NOT_FOUND(1002, "用户不存在"),
    PASSWORD_ERROR(1003, "密码错误"),

    // ===== 内容模块错误码（2xxx） =====
    ARTICLE_NOT_FOUND(2001, "文章不存在"),
    CATEGORY_NOT_FOUND(2002, "分类不存在"),
    TAG_NOT_FOUND(2003, "标签不存在"),

    // ===== 文件模块错误码（3xxx） =====
    FILE_UPLOAD_ERROR(3001, "文件上传失败"),

    // ===== 限流相关错误码（4xxx） =====
    RATE_LIMIT_EXCEEDED(4001, "请求过于频繁");

    private final int code;      // 错误码
    private final String message; // 错误描述
}
```

### 3.2 错误码设计规范

| 范围 | 模块 | 说明 |
|------|------|------|
| 200 | 通用 | 成功 |
| 400-499 | 通用 | HTTP 标准错误 |
| 500 | 通用 | 服务器内部错误 |
| 1001-1999 | 用户模块 | 用户相关错误 |
| 2001-2999 | 内容模块 | 文章/分类/标签相关错误 |
| 3001-3999 | 文件模块 | 上传/下载相关错误 |
| 4001-4999 | 限流模块 | 频率限制相关错误 |

### 3.3 为什么用枚举而不是常量？

```java
// 方式一：常量类（不推荐）
public class ErrorCodes {
    public static final int USER_NOT_FOUND_CODE = 1002;
    public static final String USER_NOT_FOUND_MSG = "用户不存在";
    // code 和 message 是分开的，容易不一致
}

// 方式二：枚举（推荐）
ErrorCode.USER_NOT_FOUND  // code 和 message 绑定在一起，不会不一致
```

枚举的优势：
1. **类型安全**：编译期检查，不会传错错误码
2. **关联属性**：code 和 message 绑定在一起
3. **可遍历**：可以通过 `ErrorCode.values()` 获取所有错误码
4. **IDE 支持**：自动补全、跳转、查找引用

## 4. PageResult 分页响应

```java
@Data
@NoArgsConstructor
@AllArgsConstructor
@Builder
public class PageResult<T> implements Serializable {

    private List<T> records;   // 当前页数据列表
    private long total;        // 总记录数
    private long page;         // 当前页码（从 1 开始）
    private long size;         // 每页条数

    // 从 MyBatis-Plus 的 IPage 对象快速构建
    public static <T> PageResult<T> from(IPage<T> page) {
        return PageResult.<T>builder()
                .records(page.getRecords())
                .total(page.getTotal())
                .page(page.getCurrent())
                .size(page.getSize())
                .build();
    }
}
```

**返回示例**：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {
    "records": [
      {"id": 1, "title": "第一篇文章", ...},
      {"id": 2, "title": "第二篇文章", ...}
    ],
    "total": 25,
    "page": 1,
    "size": 10
  }
}
```

前端分页组件可以直接使用 `total` 计算总页数，`page` 标记当前位置。

## 5. BusinessException 自定义异常

### 5.1 为什么需要自定义异常？

在业务逻辑中，很多情况并不是"程序出错"，而是"业务规则不满足"：

```java
// 场景：用户登录时用户名不存在
// 这不是程序 Bug，而是一个业务判断
if (user == null) {
    // 方式一：返回 null，让调用方判断（容易遗漏）
    return null;

    // 方式二：返回 Result.error()（每个方法都要返回 Result，不灵活）
    return Result.error(ErrorCode.USER_NOT_FOUND);

    // 方式三：抛出业务异常（推荐！由全局处理器统一处理）
    throw new BusinessException(ErrorCode.USER_NOT_FOUND);
}
```

### 5.2 实现

```java
@Getter
public class BusinessException extends RuntimeException {

    private final ErrorCode errorCode;

    // 使用预定义的错误信息
    public BusinessException(ErrorCode errorCode) {
        super(errorCode.getMessage());     // 设置异常消息
        this.errorCode = errorCode;
    }

    // 使用自定义的错误信息（覆盖 ErrorCode 的默认消息）
    public BusinessException(ErrorCode errorCode, String message) {
        super(message);
        this.errorCode = errorCode;
    }
}
```

**为什么继承 RuntimeException 而不是 Exception？**

- `RuntimeException`（非受检异常）：不需要在方法签名中声明 `throws`，代码更简洁
- `Exception`（受检异常）：调用方必须 try-catch 或声明 throws，每层都要处理，很繁琐

在 Spring 项目中，业务异常统一用非受检异常，交给全局异常处理器处理。

### 5.3 使用方式

```java
// Service 层抛出异常
public ArticleDetailVO getArticleBySlug(String slug) {
    ArticleDetailVO article = articleMapper.selectArticleBySlug(slug);
    if (article == null) {
        throw new BusinessException(ErrorCode.ARTICLE_NOT_FOUND);
        // 抛出后，GlobalExceptionHandler 会捕获并返回：
        // {"code": 2001, "message": "文章不存在", "data": null}
    }
    return article;
}

// 自定义消息
public void login(LoginDTO dto) {
    User user = userMapper.selectByUsername(dto.getUsername());
    if (user == null) {
        throw new BusinessException(ErrorCode.USER_NOT_FOUND, "用户名 " + dto.getUsername() + " 不存在");
    }
}
```

## 6. GlobalExceptionHandler 全局异常处理

### 6.1 核心注解

```java
@Slf4j                  // Lombok: 自动生成 log 对象
@RestControllerAdvice   // 全局异常处理 + 返回 JSON（= @ControllerAdvice + @ResponseBody）
public class GlobalExceptionHandler {
    // ...
}
```

`@RestControllerAdvice` 的作用：
- Spring 会扫描所有 `@RestControllerAdvice` 类
- 当任何 Controller 抛出异常时，都会被这里的方法捕获
- 返回值自动序列化为 JSON（因为包含 @ResponseBody 语义）

### 6.2 各异常处理方法详解

```java
// ===== 1. 业务异常 =====
@ExceptionHandler(BusinessException.class)
public Result<Void> handleBusinessException(BusinessException e) {
    // 业务异常用 WARN 级别（不是 Bug，只是业务规则不满足）
    log.warn("业务异常: code={}, message={}", e.getErrorCode().getCode(), e.getMessage());
    return Result.error(e.getErrorCode().getCode(), e.getMessage());
}
// 注意：这里没有 @ResponseStatus，HTTP 状态码默认是 200
// 因为业务异常的状态码在 Result.code 中表达，HTTP 状态码保持 200
// 前端通过 Result.code 判断是否成功

// ===== 2. 参数校验异常 =====
@ExceptionHandler(MethodArgumentNotValidException.class)
@ResponseStatus(HttpStatus.BAD_REQUEST)  // HTTP 400
public Result<Void> handleValidationException(MethodArgumentNotValidException e) {
    // 收集所有校验失败的消息，用分号拼接
    String errorMessage = e.getBindingResult().getFieldErrors().stream()
            .map(FieldError::getDefaultMessage)
            .collect(Collectors.joining("; "));
    log.warn("参数校验失败: {}", errorMessage);
    return Result.error(ErrorCode.BAD_REQUEST.getCode(), errorMessage);
}
// 示例返回：
// HTTP 400
// {"code": 400, "message": "标题不能为空; 内容不能为空", "data": null}

// ===== 3. 请求体解析异常 =====
@ExceptionHandler(HttpMessageNotReadableException.class)
@ResponseStatus(HttpStatus.BAD_REQUEST)  // HTTP 400
public Result<Void> handleHttpMessageNotReadableException(HttpMessageNotReadableException e) {
    log.warn("请求体解析失败: {}", e.getMessage());
    return Result.error(ErrorCode.BAD_REQUEST);
}
// 场景：请求体不是合法 JSON，或者 JSON 字段类型不匹配

// ===== 4. 权限不足异常 =====
@ExceptionHandler(AccessDeniedException.class)
@ResponseStatus(HttpStatus.FORBIDDEN)  // HTTP 403
public Result<Void> handleAccessDeniedException(AccessDeniedException e) {
    log.warn("权限不足: {}", e.getMessage());
    return Result.error(ErrorCode.FORBIDDEN);
}
// 场景：USER 角色访问 /admin/** 接口

// ===== 5. 兜底：所有未处理的异常 =====
@ExceptionHandler(Exception.class)
@ResponseStatus(HttpStatus.INTERNAL_SERVER_ERROR)  // HTTP 500
public Result<Void> handleException(Exception e) {
    // 未知异常用 ERROR 级别，需要排查
    log.error("服务器内部错误: ", e);  // 注意：这里打印了完整堆栈
    return Result.error(ErrorCode.INTERNAL_ERROR);
    // 返回给客户端的是通用错误信息，不暴露堆栈
}
```

### 6.3 异常匹配优先级

`@ExceptionHandler` 的匹配规则是**就近原则**——优先匹配最具体的异常类型：

```
抛出 BusinessException
    ↓ 匹配 handleBusinessException   ← 精确匹配

抛出 MethodArgumentNotValidException
    ↓ 匹配 handleValidationException  ← 精确匹配

抛出 NullPointerException
    ↓ 没有精确匹配
    ↓ 匹配 handleException (Exception.class)  ← 兜底匹配
```

### 6.4 安全原则：不要暴露堆栈信息

```java
// 错误示范：将异常详情暴露给客户端
@ExceptionHandler(Exception.class)
public Result<String> handleException(Exception e) {
    return Result.error(500, e.getMessage());
    // 可能返回：{"code": 500, "message": "com.mysql.jdbc.exceptions.jdbc4.MySQLSyntaxErrorException: Table 'blog.articles' doesn't exist"}
    // 这暴露了数据库类型、表名等敏感信息！
}

// 正确做法：给客户端通用信息，把详情记入日志
@ExceptionHandler(Exception.class)
public Result<Void> handleException(Exception e) {
    log.error("服务器内部错误: ", e);           // 完整堆栈记入日志（运维排查用）
    return Result.error(ErrorCode.INTERNAL_ERROR); // 客户端只看到"服务器内部错误"
}
```

## 7. 参数校验机制

### 7.1 JSR 380 校验注解

Spring Boot Starter Validation 提供了 JSR 380（Bean Validation 2.0）注解：

```java
@Data
public class ArticleDTO {

    @NotBlank(message = "Title must not be blank")
    // @NotBlank: 不能为 null 且 trim 后不能为空字符串
    // 适用于 String 类型
    @Size(max = 200, message = "Title must not exceed 200 characters")
    // @Size: 字符串长度限制
    private String title;

    private String summary;  // 可选字段，不加校验注解

    @NotBlank(message = "Content must not be blank")
    private String content;

    private String coverImage;
    private Long categoryId;
    private Integer status;
    private Integer isTop;
    private List<Long> tagIds;
}
```

```java
@Data
public class CommentDTO {

    @NotNull(message = "Article ID must not be null")
    // @NotNull: 不能为 null（适用于任意类型）
    private Long articleId;

    private Long parentId;  // 可选

    @NotBlank(message = "Nickname must not be blank")
    @Size(max = 50, message = "Nickname must not exceed 50 characters")
    private String nickname;

    @Email(message = "Email must be a valid email address")
    // @Email: 必须是合法的邮箱格式
    private String email;

    @NotBlank(message = "Content must not be blank")
    @Size(max = 500, message = "Content must not exceed 500 characters")
    private String content;
}
```

### 7.2 常用校验注解一览

| 注解 | 作用 | 适用类型 | 示例 |
|------|------|----------|------|
| `@NotNull` | 不能为 null | 任意 | `@NotNull Long id` |
| `@NotBlank` | 不为 null 且非空字符串 | String | `@NotBlank String name` |
| `@NotEmpty` | 不为 null 且不为空 | String/Collection | `@NotEmpty List tags` |
| `@Size` | 长度/大小范围 | String/Collection | `@Size(min=6, max=20)` |
| `@Min` / `@Max` | 数值范围 | 数字类型 | `@Min(1) Integer page` |
| `@Email` | 邮箱格式 | String | `@Email String email` |
| `@Pattern` | 正则匹配 | String | `@Pattern(regexp="^[a-z]+$")` |
| `@Past` / `@Future` | 日期在过去/未来 | 日期类型 | `@Past Date birthday` |

### 7.3 触发校验

在 Controller 的方法参数上加 `@Valid` 或 `@Validated`：

```java
@PostMapping("/v1/admin/articles")
public Result<Void> createArticle(
        @Valid @RequestBody ArticleDTO dto   // @Valid 触发对 dto 的校验
) {
    // 如果校验失败，Spring 自动抛出 MethodArgumentNotValidException
    // 被 GlobalExceptionHandler 捕获并返回 400 错误
    articleService.createArticle(dto, getCurrentUserId());
    return Result.success();
}
```

### 7.4 校验流程

```
客户端发送请求
    │
    ▼
Spring MVC 反序列化 JSON → ArticleDTO
    │
    ▼
检测到 @Valid 注解
    │
    ▼
Hibernate Validator 执行校验
    │
    ├── 所有字段校验通过 → 正常进入 Controller 方法
    │
    └── 有字段校验失败
        │
        ▼
    抛出 MethodArgumentNotValidException
        │
        ▼
    GlobalExceptionHandler.handleValidationException() 捕获
        │
        ▼
    收集所有错误信息，返回 Result.error(400, "Title must not be blank; Content must not be blank")
```

### 7.5 @Valid vs @Validated

| 特性 | @Valid (JSR 380) | @Validated (Spring) |
|------|------------------|---------------------|
| 来源 | `javax.validation` | `org.springframework` |
| 分组校验 | 不支持 | 支持（`@Validated(Create.class)`） |
| 嵌套校验 | 支持 | 支持 |
| 用在参数上 | 支持 | 支持 |
| 用在类上 | 不支持 | 支持（校验方法参数） |

目前我们使用 `@Valid` 即可满足需求。如果需要**分组校验**（创建和更新用不同的校验规则），再使用 `@Validated`。

## 8. 异常处理完整流程

把所有组件串联起来，看一个完整请求的异常处理流程：

```
客户端: POST /api/v1/admin/articles
Body: {"title": "", "content": ""}    ← 标题为空

    │
    ▼
Spring Security Filter Chain
    │ 验证 JWT Token... 通过
    ▼
DispatcherServlet
    │ 路由到 AdminArticleController.createArticle()
    ▼
Spring MVC 参数解析
    │ JSON → ArticleDTO
    │ 检测到 @Valid 注解
    ▼
Hibernate Validator 校验
    │ title 为空 → 不满足 @NotBlank
    │ content 为空 → 不满足 @NotBlank
    │ 抛出 MethodArgumentNotValidException
    ▼
GlobalExceptionHandler
    │ 匹配 handleValidationException 方法
    │ 收集错误信息: "Title must not be blank; Content must not be blank"
    ▼
返回 HTTP 400:
{
    "code": 400,
    "message": "Title must not be blank; Content must not be blank",
    "data": null
}
```

```
客户端: GET /api/v1/articles/non-exist-slug

    │
    ▼
Spring Security Filter Chain
    │ 公开接口，无需认证
    ▼
DispatcherServlet
    │ 路由到 ArticleController.getArticleBySlug("non-exist-slug")
    ▼
Controller 调用 Service
    │ articleService.getArticleBySlug("non-exist-slug")
    ▼
Service 业务逻辑
```

## 9. 面试常见问题

### Q1: Spring 中如何实现全局异常处理？

**答**：使用 `@RestControllerAdvice`（或 `@ControllerAdvice`）标注一个类，在其中定义 `@ExceptionHandler` 方法来处理不同类型的异常。当 Controller 中抛出异常时，Spring 会寻找匹配的 `@ExceptionHandler` 方法，按照异常类型的继承关系就近匹配。`@RestControllerAdvice` 等于 `@ControllerAdvice` + `@ResponseBody`，返回值会自动序列化为 JSON。

### Q2: @ControllerAdvice 和 @RestControllerAdvice 的区别？

**答**：`@RestControllerAdvice` = `@ControllerAdvice` + `@ResponseBody`。`@ControllerAdvice` 的 `@ExceptionHandler` 方法默认返回视图（ModelAndView），需要额外加 `@ResponseBody` 才能返回 JSON。`@RestControllerAdvice` 自动为所有方法添加 `@ResponseBody`，适合 RESTful API 项目。

### Q3: 为什么自定义异常继承 RuntimeException 而不是 Exception？

**答**：`RuntimeException` 是非受检异常（unchecked exception），不需要在方法签名中声明 `throws`，也不强制调用方 try-catch。这让代码更简洁，因为业务异常通常不需要在每一层都处理，而是直接冒泡到全局异常处理器统一处理。如果继承 `Exception`（受检异常），每个 Service 和 Controller 方法都要声明 `throws BusinessException`，非常繁琐。

### Q4: @Valid 和 @Validated 的区别？

**答**：`@Valid` 是 JSR 380 标准注解（`javax.validation`），`@Validated` 是 Spring 扩展注解。主要区别是 `@Validated` 支持分组校验（通过 `@Validated(CreateGroup.class)` 指定校验分组）。在方法参数上两者效果基本相同。如果需要在类上标注来校验 Service 层方法参数，必须用 `@Validated`。

### Q5: 为什么错误返回不应该暴露堆栈信息？

**答**：堆栈信息包含类名、方法名、行号、SQL 语句等敏感信息，攻击者可以利用这些信息发现系统漏洞（如数据库类型、表结构、框架版本等）。正确做法是：对外返回通用的错误提示（如"服务器内部错误"），对内将完整异常堆栈记入日志文件供运维排查。这也是 OWASP 安全最佳实践之一。

### Q6: 如何区分使用 @NotNull、@NotEmpty、@NotBlank？

**答**：
- `@NotNull`：只检查是否为 null，不检查空字符串。适用于 Long、Integer 等非 String 类型。`null` 不通过，`""` 通过。
- `@NotEmpty`：检查不为 null 且不为空。对 String 来说 `""` 不通过但 `" "` 通过。也可用于 Collection（不能为空集合）。
- `@NotBlank`：检查不为 null 且 trim 后不为空字符串。`" "` 也不通过。**只适用于 String 类型**。

一般原则：String 用 `@NotBlank`，数字和对象用 `@NotNull`，集合用 `@NotEmpty`。
