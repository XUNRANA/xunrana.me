## 1. 技术选型与理由

### 1.1 后端框架：Spring Boot vs 其他框架

| 框架 | 优势 | 劣势 | 适用场景 |
|------|------|------|----------|
| **Spring Boot** | 生态完善、自动配置、大厂标配 | 较重、启动慢 | 企业级应用、微服务 |
| Spring MVC | 灵活、可控性强 | 配置繁琐、需要大量 XML | 传统企业项目 |
| Vert.x | 响应式、高性能 | 生态小、学习曲线陡 | 高并发 IO 密集型 |
| Quarkus | 启动快、云原生 | 生态不如 Spring 完善 | 云原生/Serverless |

**我们选择 Spring Boot 的理由**：
1. **大厂标配**：国内互联网公司后端 Java 技术栈几乎都基于 Spring Boot
2. **生态完善**：Spring Security、Spring Data Redis、MyBatis 等都有良好的整合支持
3. **自动配置**：约定优于配置（Convention over Configuration），减少模板代码
4. **社区活跃**：遇到问题容易找到解决方案，面试常考

### 1.2 ORM 框架：MyBatis-Plus vs JPA/Hibernate

| 特性 | MyBatis-Plus | JPA/Hibernate |
|------|-------------|---------------|
| **SQL 控制** | 完全掌控 SQL | 自动生成 SQL，复杂查询不灵活 |
| **学习曲线** | 低（会 SQL 就行） | 较高（需理解 JPQL、懒加载等） |
| **性能调优** | 直观（直接优化 SQL） | 需要理解 N+1 问题、缓存策略 |
| **国内使用率** | 极高（大厂主流） | 较低（外企用得多） |
| **内置 CRUD** | BaseMapper 提供 | JpaRepository 提供 |

**我们选择 MyBatis-Plus 的理由**：
1. **国内主流**：阿里、美团、字节等大厂后端普遍使用 MyBatis 系列
2. **SQL 可控**：复杂查询直接写 SQL，性能可预测，方便调优
3. **增强功能**：MyBatis-Plus 在 MyBatis 基础上提供了 BaseMapper 内置 CRUD、LambdaQueryWrapper 链式查询、分页插件等，减少重复代码
4. **面试优势**：MyBatis 的 SQL 映射原理、缓存机制是常见面试题

### 1.3 前端框架：Vue 3 vs React

| 特性 | Vue 3 | React |
|------|-------|-------|
| **上手难度** | 低（模板语法直观） | 中（JSX 需适应） |
| **国内生态** | Element Plus、Ant Design Vue | Ant Design、Material UI |
| **状态管理** | Pinia（官方推荐） | Redux / Zustand |
| **构建工具** | Vite（Vue 核心团队开发） | CRA / Next.js |

**选择 Vue 3 的理由**：
1. **国内主流**：中小厂前端大量使用 Vue，大厂也有不少 Vue 项目
2. **学习成本低**：对后端开发者友好，模板语法接近 HTML
3. **TypeScript 支持好**：Vue 3 + Composition API 对 TS 支持很好
4. **Vite 构建快**：开发体验优秀，热更新毫秒级

### 1.4 其他技术选择

| 技术 | 选择理由 |
|------|----------|
| **JWT** | 无状态认证，适合前后端分离架构，分布式环境下不需要 Session 共享 |
| **Redis** | Token 黑名单（登出时使 Token 失效）、后续缓存热点数据、接口限流 |
| **MySQL 8.0** | 成熟稳定，支持 JSON 类型、窗口函数、CTE 等新特性 |
| **SpringDoc** | Spring Boot 3 推荐的 API 文档方案（替代 Springfox/Swagger 2） |
| **Docker** | 环境一致性，一键部署，面试加分项 |
| **Lombok** | 减少 Getter/Setter/Constructor 等模板代码，提高开发效率 |

## 2. 分层架构模式

### 2.1 经典三层架构

我们的项目采用经典的**分层架构**（Layered Architecture），也叫做三层架构：

```
┌─────────────────────────────────────────────────────────┐
│                   Client (浏览器/App)                     │
└────────────────────────┬────────────────────────────────┘
                         │  HTTP Request (JSON)
                         ▼
┌─────────────────────────────────────────────────────────┐
│  Controller 层（表现层 / Presentation Layer）              │
│                                                          │
│  职责：                                                   │
│  - 接收 HTTP 请求，解析参数                                │
│  - 参数校验（@Valid）                                      │
│  - 调用 Service 层处理业务                                 │
│  - 封装统一响应 Result<T>                                  │
│                                                          │
│  注意：Controller 不包含任何业务逻辑！                       │
└────────────────────────┬────────────────────────────────┘
                         │  调用 Service 接口
                         ▼
┌─────────────────────────────────────────────────────────┐
│  Service 层（业务层 / Business Logic Layer）               │
│                                                          │
│  职责：                                                   │
│  - 实现核心业务逻辑                                        │
│  - 事务管理（@Transactional）                              │
│  - 调用 Mapper 层进行数据操作                               │
│  - Entity ←→ VO 对象转换                                  │
│  - 跨表/跨模块的业务编排                                    │
│                                                          │
│  设计：接口 + 实现类分离（面向接口编程）                      │
└────────────────────────┬────────────────────────────────┘
                         │  调用 Mapper 接口
                         ▼
┌─────────────────────────────────────────────────────────┐
│  Mapper 层（数据访问层 / Data Access Layer / DAO 层）      │
│                                                          │
│  职责：                                                   │
│  - 与数据库交互（增删改查）                                  │
│  - 简单 CRUD 使用 BaseMapper 内置方法                      │
│  - 复杂查询使用 XML 映射文件                                │
│                                                          │
│  技术：MyBatis-Plus BaseMapper + XML Mapper               │
└────────────────────────┬────────────────────────────────┘
                         │  SQL
                         ▼
┌─────────────────────────────────────────────────────────┐
│                MySQL 8.0 数据库                           │
└─────────────────────────────────────────────────────────┘
```

### 2.2 为什么要分层？

1. **职责分离**（Single Responsibility）：每一层只做一件事，代码清晰
2. **可测试性**：可以对每一层单独进行单元测试
3. **可维护性**：修改某一层不影响其他层（只要接口不变）
4. **可替换性**：比如将来把 MyBatis-Plus 换成 JPA，只需要修改 Mapper 层

### 2.3 请求处理流程

一个完整的 HTTP 请求在项目中的处理流程：

```
客户端发起 POST /api/v1/auth/login
    │
    ▼
┌── Spring Security Filter Chain ──────────────────┐
│                                                    │
│  1. CorsFilter (跨域处理)                          │
│  2. JwtAuthFilter (JWT 认证)                       │
│     - 检查 Authorization 头                        │
│     - 验证 Token 有效性                            │
│     - 设置 SecurityContext                         │
│  3. AuthorizationFilter (鉴权)                     │
│     - 检查 URL 权限配置                            │
│     - .permitAll() / .hasRole("ADMIN")             │
│                                                    │
└────────────────────┬───────────────────────────────┘
                     │
                     ▼
            DispatcherServlet
                     │
                     ▼
        Controller (AuthController)
          │ - 接收 @RequestBody LoginDTO
          │ - @Valid 触发参数校验
          │ - 调用 authService.login(dto)
          │
          ▼
        Service (AuthServiceImpl)
          │ - 使用 AuthenticationManager 验证用户名密码
          │ - 调用 JwtTokenProvider 生成 Token
          │ - 组装 LoginVO 返回
          │
          ▼
        Mapper (UserMapper)
          │ - MyBatis-Plus selectOne() 查询用户
          │
          ▼
        MySQL 数据库
          │ - 执行 SQL: SELECT * FROM user WHERE username = ?
          │
          ▼
        返回 Result.success(loginVO)
          │
          ▼
        Spring MVC 序列化为 JSON 响应
```

## 3. Spring Boot 自动配置原理

### 3.1 核心注解 @SpringBootApplication

```java
@SpringBootApplication  // 这是一个组合注解
@MapperScan("me.xunrana.blog.mapper")
public class XunranaBlogApplication {
    public static void main(String[] args) {
        SpringApplication.run(XunranaBlogApplication.class, args);
    }
}
```

`@SpringBootApplication` 实际上是三个注解的组合：

```java
@SpringBootConfiguration    // 等同于 @Configuration，标记为配置类
@EnableAutoConfiguration    // 核心！开启自动配置
@ComponentScan              // 扫描当前包及子包下的 @Component/@Service/@Controller 等
```

### 3.2 自动配置流程

```
1. 应用启动
   │
   ▼
2. @EnableAutoConfiguration 生效
   │
   ▼
3. 读取 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports
   │  (Spring Boot 3.x 使用此文件，2.x 使用 spring.factories)
   │
   ▼
4. 加载所有候选的 AutoConfiguration 类
   │  例如：DataSourceAutoConfiguration, RedisAutoConfiguration,
   │       SecurityAutoConfiguration, WebMvcAutoConfiguration ...
   │
   ▼
5. 根据条件注解（@ConditionalOnXxx）判断是否生效
   │  @ConditionalOnClass      → classpath 下是否有某个类
   │  @ConditionalOnProperty   → 配置文件中是否有某个属性
   │  @ConditionalOnBean       → 容器中是否有某个 Bean
   │  @ConditionalOnMissingBean → 容器中是否缺少某个 Bean
   │
   ▼
6. 满足条件的配置类创建对应的 Bean，注入 IoC 容器
```

**举例**：当我们在 `pom.xml` 中添加了 `spring-boot-starter-data-redis`：
- Spring Boot 发现 classpath 下有 Redis 相关的类
- `RedisAutoConfiguration` 的条件满足
- 自动创建 `RedisConnectionFactory`、`RedisTemplate`、`StringRedisTemplate` 等 Bean
- 自动读取 `spring.data.redis.host`、`spring.data.redis.port` 等配置

**重要**：我们在 `RedisConfig.java` 中自定义了 `RedisTemplate<String, Object>` Bean，此时 Spring Boot 的 `@ConditionalOnMissingBean` 检测到已经存在同类型 Bean，就不会再创建默认的，**用户自定义配置优先级更高**。

### 3.3 @MapperScan 的作用

```java
@MapperScan("me.xunrana.blog.mapper")
```

这个注解告诉 MyBatis-Plus 去扫描 `me.xunrana.blog.mapper` 包下的所有接口，并为它们**动态生成代理实现类**注入到 Spring 容器中。虽然我们的 Mapper 接口上也标注了 `@Mapper`，但 `@MapperScan` 可以批量扫描，避免每个 Mapper 都要加 `@Mapper`。

## 4. pom.xml 依赖详解

### 4.1 父项目 (Parent POM)

```xml
<parent>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-parent</artifactId>
    <version>3.2.3</version>
</parent>
```

**作用**：
- 统一管理 Spring Boot 相关依赖的版本号（BOM: Bill of Materials）
- 配置 Java 编译版本、编码、插件等
- 子项目引入 Spring Boot 的 starter 时不需要指定版本号

### 4.2 各依赖作用

```xml
<!-- ==================== Spring Boot Starters ==================== -->

<!-- spring-boot-starter-web -->
<!-- 提供：Spring MVC、内嵌 Tomcat、Jackson JSON 序列化 -->
<!-- 有了它，项目就是一个 Web 应用，可以处理 HTTP 请求 -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
</dependency>

<!-- spring-boot-starter-security -->
<!-- 提供：Spring Security 安全框架 -->
<!-- 认证（你是谁）+ 授权（你能做什么）-->
<!-- 引入后，所有接口默认需要认证才能访问 -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-security</artifactId>
</dependency>

<!-- spring-boot-starter-data-redis -->
<!-- 提供：RedisTemplate、Lettuce 连接池 -->
<!-- 用于 Token 黑名单、后续做缓存和限流 -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-redis</artifactId>
</dependency>

<!-- spring-boot-starter-validation -->
<!-- 提供：JSR 380 参数校验（@NotBlank、@Size、@Email 等） -->
<!-- 配合 @Valid 注解实现请求参数自动校验 -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-validation</artifactId>
</dependency>

<!-- ==================== MyBatis-Plus ==================== -->

<!-- mybatis-plus-spring-boot3-starter -->
<!-- 注意：Spring Boot 3.x 要用 boot3-starter，2.x 用 boot-starter -->
<!-- 提供：BaseMapper CRUD、LambdaQueryWrapper、分页插件等 -->
<dependency>
    <groupId>com.baomidou</groupId>
    <artifactId>mybatis-plus-spring-boot3-starter</artifactId>
    <version>${mybatis-plus.version}</version>
</dependency>

<!-- ==================== MySQL Driver ==================== -->

<!-- mysql-connector-j -->
<!-- MySQL 的 JDBC 驱动程序 -->
<!-- scope=runtime：编译时不需要，运行时才需要（面向接口编程） -->
<dependency>
    <groupId>com.mysql</groupId>
    <artifactId>mysql-connector-j</artifactId>
    <scope>runtime</scope>
</dependency>

<!-- ==================== JWT ==================== -->

<!-- jjwt 三件套：api + impl + jackson -->
<!-- api：接口定义（编译时依赖） -->
<!-- impl：核心实现（运行时依赖） -->
<!-- jackson：JSON 序列化支持（运行时依赖） -->
<!-- 为什么分成三个？模块化设计，api 和 impl 分离 -->
<dependency>
    <groupId>io.jsonwebtoken</groupId>
    <artifactId>jjwt-api</artifactId>
    <version>${jjwt.version}</version>
</dependency>
<dependency>
    <groupId>io.jsonwebtoken</groupId>
    <artifactId>jjwt-impl</artifactId>
    <version>${jjwt.version}</version>
    <scope>runtime</scope>
</dependency>
<dependency>
    <groupId>io.jsonwebtoken</groupId>
    <artifactId>jjwt-jackson</artifactId>
    <version>${jjwt.version}</version>
    <scope>runtime</scope>
</dependency>

<!-- ==================== 工具类 ==================== -->

<!-- Lombok：编译期注解处理器 -->
<!-- @Data = @Getter + @Setter + @ToString + @EqualsAndHashCode -->
<!-- @Builder = 构建者模式 -->
<!-- @RequiredArgsConstructor = final 字段的构造函数（替代 @Autowired） -->
<!-- optional=true：不传递给依赖此项目的其他项目 -->
<dependency>
    <groupId>org.projectlombok</groupId>
    <artifactId>lombok</artifactId>
    <optional>true</optional>
</dependency>

<!-- SpringDoc OpenAPI：自动生成 API 文档 -->
<!-- 替代已停止维护的 Springfox/Swagger 2 -->
<!-- 访问 /swagger-ui.html 查看交互式 API 文档 -->
<dependency>
    <groupId>org.springdoc</groupId>
    <artifactId>springdoc-openapi-starter-webmvc-ui</artifactId>
    <version>${springdoc.version}</version>
</dependency>

<!-- Hutool：Java 工具类库 -->
<!-- 字符串处理、日期处理、加密解密、HTTP 客户端等 -->
<dependency>
    <groupId>cn.hutool</groupId>
    <artifactId>hutool-all</artifactId>
    <version>${hutool.version}</version>
</dependency>
```

### 4.3 Maven Build 插件

```xml
<build>
    <plugins>
        <plugin>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-maven-plugin</artifactId>
            <configuration>
                <excludes>
                    <!-- 打包时排除 Lombok（它只在编译期有用） -->
                    <exclude>
                        <groupId>org.projectlombok</groupId>
                        <artifactId>lombok</artifactId>
                    </exclude>
                </excludes>
            </configuration>
        </plugin>
    </plugins>
</build>
```

`spring-boot-maven-plugin` 的作用是将项目打包为**可执行的 Fat JAR**（包含所有依赖和内嵌 Tomcat），可以直接 `java -jar blog-backend.jar` 运行。

## 5. application.yml 配置详解

### 5.1 主配置文件 application.yml

```yaml
spring:
  profiles:
    active: dev   # 激活 dev 环境配置
                  # Spring Boot 会加载 application.yml + application-dev.yml
                  # 生产环境改为 prod，加载 application.yml + application-prod.yml

server:
  port: 8080           # 应用监听端口
  servlet:
    context-path: /api  # 上下文路径，所有 API 前缀为 /api
                        # 例如 @GetMapping("/v1/articles") 实际路径为 /api/v1/articles
```

### 5.2 开发环境配置 application-dev.yml

```yaml
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/xunrana_blog?useUnicode=true&characterEncoding=utf-8&useSSL=false&serverTimezone=Asia/Shanghai&allowPublicKeyRetrieval=true
    # URL 参数说明：
    # useUnicode=true & characterEncoding=utf-8   → 支持中文
    # useSSL=false                                 → 本地开发不需要 SSL
    # serverTimezone=Asia/Shanghai                 → 时区设为东八区
    # allowPublicKeyRetrieval=true                 → MySQL 8 密码认证需要
    username: root
    password: root123
    driver-class-name: com.mysql.cj.jdbc.Driver  # MySQL 8 的驱动类

  data:
    redis:
      host: localhost
      port: 6379
      database: 0          # Redis 数据库编号（0-15）
      timeout: 10000ms     # 连接超时时间

  servlet:
    multipart:
      max-file-size: 5MB      # 单个文件最大 5MB
      max-request-size: 5MB   # 整个请求最大 5MB

# JWT 配置
jwt:
  secret: xunrana-blog-dev-secret-key-must-be-at-least-64-characters-long-for-hmac-sha256-signing
  # 密钥长度必须 >= 64 字符（512 位），HMAC-SHA256 要求至少 256 位
  access-expiration: 1800000    # AccessToken 过期时间：30 分钟 (30 * 60 * 1000)
  refresh-expiration: 604800000 # RefreshToken 过期时间：7 天 (7 * 24 * 60 * 60 * 1000)

# MyBatis-Plus 配置
mybatis-plus:
  mapper-locations: classpath:mapper/**/*.xml  # XML 映射文件位置
  global-config:
    db-config:
      logic-delete-field: deleted           # 逻辑删除字段名
      logic-delete-value: 1                 # 删除标记值
      logic-not-delete-value: 0             # 未删除标记值
  configuration:
    log-impl: org.apache.ibatis.logging.stdout.StdOutImpl  # 开发环境打印 SQL
    map-underscore-to-camel-case: true      # 下划线转驼峰：created_at → createdAt

# 文件上传路径
file:
  upload-dir: ./uploads/

# 日志级别
logging:
  level:
    me.xunrana.blog: debug                    # 项目日志级别
    org.springframework.security: debug       # Security 日志（开发时调试用）
```

### 5.3 生产环境配置 application-prod.yml

```yaml
spring:
  datasource:
    url: jdbc:mysql://${DB_HOST}:${DB_PORT}/${DB_NAME}?...
    username: ${DB_USERNAME}       # 通过环境变量注入
    password: ${DB_PASSWORD}       # 绝不在代码中硬编码生产密码！

  data:
    redis:
      host: ${REDIS_HOST}
      password: ${REDIS_PASSWORD:}  # 冒号后为默认值（空字符串）

jwt:
  secret: ${JWT_SECRET}            # 生产环境使用高强度随机密钥

logging:
  level:
    me.xunrana.blog: info          # 生产环境降低日志级别
    org.springframework.security: warn
```

**重要设计**：生产配置使用 `${环境变量}` 语法，敏感信息通过环境变量（或 Docker secrets）注入，不写在代码里。

### 5.4 多环境配置的加载顺序

```
application.yml         ← 公共配置（总是加载）
    ↓ 合并
application-{profile}.yml  ← 环境特定配置（覆盖公共配置中的同名属性）
    ↓
最终配置
```

通过 `spring.profiles.active=dev` 或启动参数 `--spring.profiles.active=prod` 切换环境。

## 6. 包结构设计哲学

```
me.xunrana.blog/
├── XunranaBlogApplication.java   # 启动类，放在根包下
│                                  # @ComponentScan 默认扫描当前包及子包
│
├── config/                        # 配置类 —— "项目怎么配置"
│   ├── SecurityConfig.java        # Spring Security 安全规则
│   ├── RedisConfig.java           # Redis 序列化方式
│   ├── MyBatisPlusConfig.java     # 分页插件等 MyBatis-Plus 配置
│   └── CorsConfig.java           # 跨域访问配置
│
├── security/                      # 安全认证 —— "JWT 相关逻辑"
│   ├── JwtTokenProvider.java      # Token 生成、解析、验证
│   ├── JwtAuthFilter.java         # 请求拦截与认证
│   └── UserDetailsServiceImpl.java # 从 DB 加载用户信息
│
├── common/                        # 公共组件 —— "全局通用的东西"
```

### 设计原则

1. **按职能分包**（Package by Layer）：同一层的代码放在一起
2. **启动类放根包**：确保 `@ComponentScan` 能扫描到所有子包
3. **model 独立**：Entity、DTO、VO 放在 model 下，清晰区分数据角色
4. **接口与实现分离**：Service 定义接口，impl 子包放实现类（面向接口编程）
5. **公开与管理员分离**：controller/admin 子包放管理后台接口

## 7. Entity vs DTO vs VO 数据流转

### 7.1 三种对象的区别

| 对象 | 全称 | 方向 | 用途 | 示例 |
|------|------|------|------|------|
| **DTO** | Data Transfer Object | 前端 → 后端 | 接收请求参数 | `LoginDTO`、`ArticleDTO` |
| **Entity** | 实体 | 后端内部 | 映射数据库表 | `User`、`Article` |
| **VO** | View Object | 后端 → 前端 | 返回响应数据 | `LoginVO`、`ArticleVO` |

### 7.2 数据流转图

```
前端                           后端                              数据库
┌──────┐                ┌─────────────────────┐              ┌──────┐
│      │   LoginDTO     │  Controller         │              │      │
│      │  {username,    │    │                │              │      │
│      │   password}    │    ▼                │              │      │
│      │ ──────────────►│  Service            │   Entity     │      │
│      │                │    │ DTO → Entity   │  (User)      │      │
│      │                │    ▼                │ ────────────► │      │
│      │                │  Mapper             │   SQL         │      │
│      │                │    │                │              │      │
│      │                │    ▼                │  Entity      │      │
│      │   LoginVO      │  Service            │ ◄──────────── │      │
│      │  {accessToken, │    │ Entity → VO    │              │      │
│      │   refreshToken,│    ▼                │              │      │
│      │   user: {...}} │  Controller         │              │      │
│      │ ◄──────────────│    │ Result.success(vo)            │      │
│      │                │                     │              │      │
└──────┘                └─────────────────────┘              └──────┘
```

### 7.3 为什么要区分三种对象？

**为什么不直接用 Entity？**

1. **安全性**：Entity 包含所有字段（如 `password`），不能直接返回给前端
2. **灵活性**：前端需要的数据格式可能与数据库不一致（如 tags 列表需要关联查询）
3. **解耦**：数据库结构变化不应该直接影响 API 返回格式

**实际例子**：

```java
// Entity - 对应数据库 user 表的所有字段
public class User {
    private Long id;
    private String username;
    private String password;   // 密码！绝不能返回给前端
    private String nickname;
    private String avatar;
    private String email;
    private Integer role;
    private Integer status;    // 内部状态，前端不需要
    private LocalDateTime createdAt;
    private LocalDateTime updatedAt;
}

// DTO - 登录时只需要用户名和密码
public class LoginDTO {
    @NotBlank
    private String username;
    @NotBlank @Size(min=6, max=20)
    private String password;
}

// VO - 返回给前端的用户信息（不含密码和内部状态）
public class UserVO {
    private Long id;
    private String username;
    private String nickname;
    private String avatar;
    private String email;
    private Integer role;
    private LocalDateTime createdAt;
}
```

### 7.4 对象转换时机

- **DTO → Entity**：在 Service 层，创建/更新数据时
- **Entity → VO**：在 Service 层，查询数据后返回前

```java
// Service 层示例
public void createArticle(ArticleDTO dto, Long authorId) {
    // DTO → Entity
    Article article = new Article();
    article.setTitle(dto.getTitle());
    article.setContent(dto.getContent());
    article.setAuthorId(authorId);
    // ... 设置其他字段
    articleMapper.insert(article);  // Entity 存入数据库
}

public ArticleVO getArticle(Long id) {
    Article article = articleMapper.selectById(id);  // 从数据库取 Entity
    // Entity → VO
    ArticleVO vo = new ArticleVO();
    vo.setId(article.getId());
    vo.setTitle(article.getTitle());
    // ... 复制需要返回的字段（不包含内部字段）
    return vo;
}
```

## 8. 面试常见问题

### Q1: Spring Boot 的自动配置原理是什么？

**答**：Spring Boot 通过 `@EnableAutoConfiguration` 注解，利用 SPI 机制加载 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` 文件中列出的所有自动配置类。这些配置类通过 `@ConditionalOnClass`、`@ConditionalOnProperty`、`@ConditionalOnMissingBean` 等条件注解判断是否生效。比如当 classpath 下有 `RedisTemplate` 类且没有用户自定义的 `RedisTemplate` Bean 时，`RedisAutoConfiguration` 才会自动创建默认的 `RedisTemplate`。

### Q2: 为什么选择 MyBatis-Plus 而不是 JPA？

**答**：MyBatis-Plus 允许直接编写 SQL，对 SQL 有完全控制权，便于性能调优和处理复杂查询。JPA 虽然抽象程度更高，但在复杂关联查询和性能优化方面不够灵活，而且存在 N+1 查询、懒加载等需要特别注意的坑。在国内大厂（阿里、美团、字节等），MyBatis 系列是主流选择。

### Q3: 项目为什么要分层？Controller/Service/Mapper 各层职责是什么？

**答**：分层是为了实现关注点分离。Controller 层负责接收请求、参数校验和响应封装，不包含业务逻辑；Service 层负责核心业务处理和事务管理；Mapper 层负责与数据库交互。这样设计的好处是每一层可以独立测试和替换，提高代码的可维护性和可测试性。

### Q4: Entity、DTO、VO 有什么区别？为什么不直接用 Entity？

**答**：Entity 映射数据库表结构，DTO 用于接收前端请求参数，VO 用于构建返回给前端的数据。不直接使用 Entity 的原因：一是安全（Entity 可能包含密码等敏感字段），二是灵活（前端需要的数据格式可能与数据库不同），三是解耦（数据库表结构的变化不应直接影响 API 接口格式）。

### Q5: Spring Boot 多环境配置是怎么实现的？

**答**：通过 `spring.profiles.active` 属性指定当前激活的环境。Spring Boot 会先加载 `application.yml`（公共配置），然后加载 `application-{profile}.yml`（环境特定配置），后者会覆盖前者中的同名属性。生产环境配置使用 `${ENV_VAR}` 语法从环境变量读取敏感信息（如数据库密码），避免硬编码。

### Q6: Maven 中的 scope 有哪些？runtime 是什么意思？

**答**：Maven 的 scope 主要有 compile（默认，编译运行都需要）、runtime（编译时不需要，运行时需要，如 JDBC 驱动）、provided（编译时需要但运行环境已提供，如 Servlet API）、test（只在测试时使用）。我们的 MySQL 驱动使用 runtime scope，因为代码中通过 JDBC 接口编程，不直接依赖具体驱动类。
